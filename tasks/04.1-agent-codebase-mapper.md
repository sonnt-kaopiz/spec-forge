# 04.1 — Agent: codebase-mapper

**Phase**: 1 - Foundation
**Priority**: High
**Depends on**: 01
**Plan reference**: forge:map-codebase workflow

## Description

Define the `codebase-mapper` subagent. This agent is instantiated in parallel — **one per service** — during `forge:map-codebase`. Each instance receives a single service root and an output path, explores that service comprehensively across all four domains (stack, integration, architecture, structure), and **writes a single raw findings file directly**. It reports back only confirmation and line count.

> **Redesign note** (updated 2026-04-06): The original design spawned one agent per focus area (4 agents per service). The revised `forge:map-codebase` command (task 04.2) spawns one agent per service and aggregates domain documents in the orchestrator after all scouts complete. The agent prompt must be updated to match: `focus_area` input is removed; the agent covers all four domains in one pass and writes a structured raw-findings file (not a polished per-domain document).

## Deliverables

- [x] `agents/codebase-mapper.md` — rewritten to scout mode:
  - Input: `service_root`, `output_path` (no `focus_area`)
  - Behavior: explore the service for all four domains in one pass:
    - **stack**: language detection via manifest files (composer.json, Gemfile, package.json, go.mod, etc.), runtimes, frameworks, dependencies, environment config
    - **integration**: inter-service communication (HTTP clients, queues, events), databases (connections, schemas), authentication mechanisms
    - **architecture**: this service's system role and positioning (bounded context, topology, data flow) — not internal code patterns
    - **structure**: directory layout, key locations (routes, controllers, models, tests), naming conventions
  - Output behavior: writes structured raw-findings markdown to `output_path`, then returns `{ status, lines, path, service, reason? }`
  - Self-contained: includes all exploration instructions without relying on external context

## Output Document Format

Each scout agent writes a single file covering all four domains:

```
# Scout — <Service Name>
_Generated: <timestamp>_

## Stack
Structured findings: language/version, framework/version, dependencies, env config, build tooling.

## Integration
Structured findings: outbound HTTP calls, queues/events, databases, auth, third-party integrations.

## Architecture
Structured findings: service purpose, domain ownership, system topology, data flow summary.

## Structure
Structured findings: directory tree (key paths only), entry points, naming conventions.

## Key Files
Bulleted list of the most important files across all focus areas.
```

Output size target: 100–200 lines. Hard cap: 200 lines. Condense rather than exceed.

## Acceptance Criteria

- Agent prompt is self-contained (no CLAUDE.md dependency)
- Agent accepts `service_root` and `output_path` inputs (no `focus_area`)
- Agent covers all four domains in a single exploration pass
- Agent writes the output file directly using Write tool
- Agent returns structured confirmation `{ status, lines, path, service }` after writing
- Output is structured raw findings suitable for downstream aggregation by the orchestrator

## Decisions

> Discussed on 2026-04-06

### Exploration budget & output cap
No explicit exploration cap in the prompt — trust the Sonnet agent's judgement and Claude Code's natural limits rather than hard file-read budgets that could miss important context on large repos. However, output size is hard-capped at ~200 lines per document to keep downstream consumers (index.md, later research agents) light and predictable. The prompt should instruct the agent to target 100–200 lines and truncate/condense findings rather than exceed the cap.

### Focus area scope boundaries
Stack exclusively owns dependency manifests and environment config (composer.json, package.json, .env.example, config files). The `architecture` focus area is scoped to **system architecture** — this service's role and positioning within the broader system (bounded context, system-level patterns, how it fits into overall service topology and data flow) — **not** per-service internal code patterns like MVC/DDD/service container usage. Internal code layout and file organization belong to the `structure` focus area. Framework is named lightly by both Stack (as a runtime fact: "Laravel 11") and Architecture (for context: "Laravel-based edge service in the X system"). Each focus area section in the prompt must include explicit `DO cover` and `DO NOT cover (belongs to <X>)` lists so parallel instances cannot produce overlapping content.

### Return contract & failures
Return shape: `{status, lines, path, service, reason?}`.
- `service` = service name derived from `service_root` directory name or manifest `name` field.
- `lines` = total line count of the written file (`wc -l` semantics, including blank lines and headers). Easy for the agent to compute and for the orchestrator to display in `index.md`.
- Valid `status` values:
  - `done` — fully explored all four domains and written.
  - `partial` — file written but exploration was incomplete for one or more domains (hit a budget or blocker). `reason` required.
  - `failed` — could not produce meaningful content, but the agent still writes a minimal stub file containing the failure reason so the path always exists for the orchestrator. `reason` required.
- The file at `output_path` is **always** written, even on failure, so `#04.2`'s orchestrator can rely on path existence without conditional checks.
