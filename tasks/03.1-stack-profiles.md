# 03.1 — Built-in Stack Profiles

**Phase**: 1 - Foundation
**Priority**: High
**Depends on**: 03
**Plan reference**: forge.yaml stacks section

## Description

Validate and document the built-in stack profiles shipped in `forge.yaml`. Each profile must be accurate, testable, and serve as a reference for users building custom profiles.

## Deliverables

- [x] Verify all built-in profiles in `forge.yaml` under the `stacks` key:
  - `laravel` — PHP 8.x, Laravel 11, composer.json, php artisan test, phpstan, pint
  - `yii2` — PHP 8.x, Yii2 2.0, composer.json, phpunit, phpstan, php-cs-fixer
  - `rails` — Ruby, Rails 7, Gemfile, bundle exec rails test, rubocop
  - `django` — Python, Django 5, pyproject.toml, python manage.py test, ruff
  - `express` — JavaScript, Express 5, package.json, npm test, eslint, prettier
  - `springboot` — Java, Spring Boot 3, pom.xml, mvn test, checkstyle, spotless
  - `go` — Go 1.22+, stdlib, go.mod, go test ./..., golangci-lint, gofmt
- [x] Each profile must include: `language`, `framework`, `version`, `manifest`, `paths`, `verification.test`, `verification.analyze`, `verification.format`
- [x] Add a note in `forge.yaml` explaining how to add custom profiles

## Acceptance Criteria

- All built-in profiles have the correct manifest file name for their ecosystem
- Verification commands are real, documented CLI commands for each tool
- Path conventions match the actual framework's default directory structure
- A user can add a custom profile by copying an existing profile block and editing it

## Notes

- These profiles are documentation as much as they are config — they also inform agents about where to look when exploring a codebase
- Keep profile commands generic (no path assumptions beyond the profile's `paths` block)
- Users with non-standard directory layouts should override `paths` in `forge-service.yaml`

## Decisions

> Discussed on 2026-04-06

### paths schema consistency
All profiles must include the 6 standard keys (`models`, `services`, `controllers`, `migrations`, `tests_unit`, `tests_feature`). Profiles may add extra profile-specific keys on top (e.g. `controllers_backend` for yii2), but the standard 6 are required so that downstream agents (codebase-researcher and others) can read paths by a consistent key name. For yii2 specifically, the `controllers` key should point to `api/controllers`, reflecting an API-first yii2 layout; the additional `controllers_frontend` and `controllers_backend` keys are kept as extras.

### analyze.config optional or required
The `config` key is optional on all three verification steps (`test`, `analyze`, `format`) — only add it when the tool actually uses configuration values; omit it otherwise. The key design point is that the `config` pattern is available for all three steps (not only `analyze`), so downstream code that reads config values should check any of the three steps, not just `analyze`. Profiles that currently omit `config` (rails, django, express, springboot, go) retain that behavior.

### custom profile instructions format
Add a YAML comment block directly above the `stacks:` key in `forge.yaml`. The comment should cover three things: (1) how to copy an existing profile block and rename it for a new stack, (2) which keys are required (`language`, `framework`, `version`, `manifest`, `paths` with the 6 standard subkeys, `verification.test.command`, `verification.analyze.command`, `verification.format.command`) vs optional (e.g. `filter_flag`, `config`), and (3) how to place a custom profile in `forge-service.yaml` instead of `forge.yaml` for service-local stacks.

### what "verify" means
Verification requires live execution: spin up a minimal project structure per stack and run the declared commands to confirm they function correctly (e.g. `php artisan test` on a minimal Laravel app, `go test ./...` on a minimal Go module). The deliverable is two things: (1) a corrected `forge.yaml` with any inaccuracies fixed, and (2) a new profile reference document in `docs/` that agents can read to understand each stack's conventions, path layout rationale, and tool choices.
