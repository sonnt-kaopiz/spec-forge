# 04.2 — Command: /forge:map-codebase

**Phase**: 1 - Foundation
**Priority**: High
**Depends on**: 04.1
**Plan reference**: forge:map-codebase workflow

## Description

Create the `/forge:map-codebase` slash command. This command maps the entire workspace (all services) by default, or a specific subset of services if the user names them. It operates in three phases: discovery (find services), parallel scouting (one `codebase-mapper` agent per service), and aggregation (synthesize per-service findings into four system-level domain documents).

The command is stateless — it does not interact with `state.yaml` and can be run at any time independently of the task workflow.

## Workflow

```
Phase 1 — Discovery
  └─ Resolve workspace root
  └─ Scan for services (subdirs with manifest files or forge-service.yaml)
  └─ Filter to user-specified services if provided

Phase 2 — Scouting (all agents run in parallel)
  ├─ codebase-mapper → service-a.md
  ├─ codebase-mapper → service-b.md
  └─ codebase-mapper → service-n.md

Phase 3 — Aggregation (orchestrator writes final docs)
  ├─ stack.md        (tech stack across all services)
  ├─ integration.md  (cross-service communication map)
  ├─ architecture.md (system architecture and data flow)
  ├─ structure.md    (workspace + service folder layout)
  └─ index.md        (summary with links and line counts)
```

## Deliverables

- [x] `commands/forge/map-codebase.md` — command prompt that:
  1. Resolves `workspace_root`:
     - If `forge-service.yaml` exists in cwd, read its `workspace_root` field
     - Otherwise default to cwd
  2. Discovers services:
     - Scan immediate subdirectories of `workspace_root` for manifest files (`composer.json`, `Gemfile`, `package.json`, `go.mod`, `pyproject.toml`, `pom.xml`, `Cargo.toml`) or `forge-service.yaml`
     - If the user passed service name arguments, filter to only those directories
     - Skip directories that contain none of the above signals
  3. Ensures `.ai-workflow/codebase/scouts/` directory exists under `workspace_root`
  4. Spawns one `codebase-mapper` agent per discovered service **in parallel**, each with:
     - `service_root`: absolute path to the service directory
     - `output_path`: `<workspace_root>/.ai-workflow/codebase/scouts/<service-name>.md`
  5. Collects results `{ status, lines, path, service }` from all scout agents
  6. Reads all successfully written scout files
  7. Synthesizes and writes the four domain documents (see templates below)
  8. Writes `index.md` (see format below)
  9. Reports completion to the user: files written, total lines, any agent failures

## Output Files

```
<workspace_root>/.ai-workflow/codebase/
├── index.md           ← orchestrator writes this (summary + links)
├── stack.md           ← aggregated tech stack across all services
├── integration.md     ← aggregated cross-service communication
├── architecture.md    ← aggregated system architecture
├── structure.md       ← workspace and service folder structure
└── scouts/
    ├── <service-a>.md ← raw per-service findings (written by agents)
    ├── <service-b>.md
    └── ...
```

## Design Constraints

- The orchestrator reads scout files directly to write domain documents — it does not spawn additional agents for aggregation
- If a scout agent fails, its service is noted in `index.md` with failure reason; aggregation continues with remaining scouts
- If all scouts fail, the command reports failure and writes no domain documents
- Command is idempotent: re-running overwrites all files in `.ai-workflow/codebase/` (including scouts)
- No `state.yaml` involvement — this command is stateless and can run at any time
- The `scouts/` directory holds intermediate files; consumers should read the domain documents, not the scout files

## Acceptance Criteria

- Command resolves workspace root correctly (from `forge-service.yaml` or cwd fallback)
- Service discovery finds all subdirectories with manifest files or `forge-service.yaml`
- User-specified service filtering works (e.g. `/forge:map-codebase user-service order-service`)
- All scout agents run in parallel (not sequentially)
- Each scout agent writes its file independently to `scouts/<service-name>.md`
- All four domain documents are written after scouts complete
- `index.md` includes per-service scout status and links
- Command output tells the user exactly which files were created, line counts, and any failures
- Command works from any workspace root, not just spec-forge itself
- Partial results (some scouts failed) produce valid output with available data
