# Stack Profiles Reference

This document describes each built-in stack profile shipped in `forge.yaml`. It is intended for agents (codebase-researcher, solution-architect, phase-planner) that need to understand a service's directory layout, conventions, and toolchain before analysing or generating code.

## How to Read This Document

Each profile section covers:
- **Directory layout** — where the key source directories live and why
- **Verification tools** — what each tool does and how to invoke it
- **Agent navigation hints** — how to interpret path entries when exploring the codebase

All profiles share a common paths schema with six standard keys:

| Key | Meaning |
|-----|---------|
| `models` | Domain model definitions (classes, structs, Eloquent models, Active Records, etc.) |
| `services` | Business logic layer (service objects, use-case handlers) |
| `controllers` | HTTP request entry points (controllers, views, handlers) |
| `migrations` | Database schema migrations |
| `tests_unit` | Unit tests (fast, no I/O, test single units of logic) |
| `tests_feature` | Integration / feature tests (full stack, database, HTTP) |

Paths that contain `*` or `**` are glob patterns; agents should expand them when exploring.

## laravel

**Language**: PHP 8.x | **Framework**: Laravel 11 | **Manifest**: `composer.json`

### Directory Layout

```
app/
  Models/          ← Eloquent models (one class per file, PascalCase)
  Services/        ← Service classes (not a Laravel default; project convention)
  Http/
    Controllers/   ← HTTP controllers (resourceful or invokable)
database/
  migrations/      ← Laravel migration files (timestamp-prefixed)
tests/
  Unit/            ← PHPUnit unit tests
  Feature/         ← PHPUnit feature/HTTP tests
composer.json      ← dependency manifest
```

### Verification Tools

| Step | Command | Notes |
|------|---------|-------|
| test | `php artisan test` | Wraps PHPUnit; use `--filter <name>` to run one test |
| analyze | `vendor/bin/phpstan analyse` | Static analysis; `config.level` controls strictness (0–9) |
| format | `vendor/bin/pint` | Laravel's opinionated PHP formatter (wraps PHP-CS-Fixer) |

### Agent Notes

- `app/Services/` is not generated by default. If absent, agents should look for service-layer logic inside controllers or dedicated action classes (`app/Actions/`).
- Migration filenames are timestamp-prefixed: `2024_01_01_000000_create_users_table.php`.
- Test namespaces mirror the directory: `Tests\Unit\`, `Tests\Feature\`.

## yii2

**Language**: PHP 8.x | **Framework**: Yii2 2.0 | **Manifest**: `composer.json`

### Directory Layout

```
common/
  models/          ← shared Active Record models (used by all apps)
  services/        ← shared service classes
api/
  controllers/     ← REST API controllers (standard key: controllers)
frontend/
  controllers/     ← Web frontend controllers (extra key: controllers_frontend)
backend/
  controllers/     ← Admin backend controllers (extra key: controllers_backend)
console/
  migrations/      ← Yii2 database migrations
tests/
  unit/            ← PHPUnit unit tests
  functional/      ← Codeception functional tests
composer.json
```

### Verification Tools

| Step | Command | Notes |
|------|---------|-------|
| test | `vendor/bin/phpunit` | Use `--filter <name>` for a single test |
| analyze | `vendor/bin/phpstan analyse` | `config.level` 6 is standard for Yii2 codebases |
| format | `vendor/bin/php-cs-fixer fix` | PSR-12 style enforcement |

### Agent Notes

- Yii2 Advanced Template has three apps: `frontend`, `backend`, `api`. The standard `controllers` key points to `api/controllers` (REST entry point). Use the extra keys `controllers_frontend` and `controllers_backend` to navigate web and admin controllers.
- Migrations live in `console/migrations/` and are numbered, not timestamped.
- `common/` holds code shared across all apps; always check here before assuming a model is app-specific.

## rails

**Language**: Ruby | **Framework**: Rails 7 | **Manifest**: `Gemfile`

### Directory Layout

```
app/
  models/          ← Active Record models
  services/        ← service objects (not Rails default; project convention)
  controllers/     ← Action controllers
db/
  migrate/         ← Rails migrations (timestamp-prefixed)
test/
  models/          ← Minitest model unit tests
  integration/     ← Minitest integration tests
Gemfile            ← dependency manifest
```

### Verification Tools

| Step | Command | Notes |
|------|---------|-------|
| test | `bundle exec rails test` | Runs Minitest; use `-n <name>` to filter by test name |
| analyze | `bundle exec rubocop --format json` | JSON output for machine parsing; covers style + lint |
| format | `bundle exec rubocop -a` | Auto-corrects safe offences; `-A` for unsafe corrections |

### Agent Notes

- `app/services/` is a common convention but not generated by Rails. If absent, look for business logic in concerns (`app/concerns/`) or POROs adjacent to models.
- Rails migrations are in `db/migrate/` (not `db/migrations/`).
- Minitest filter flag is `-n` (short form); it accepts a test name string or `/regex/`.
- RuboCop serves as both the linter (analyze step) and formatter (format step) with different flags.

## django

**Language**: Python | **Framework**: Django 5 | **Manifest**: `pyproject.toml`

### Directory Layout

Django projects are composed of multiple apps, each with its own directory. Paths use glob patterns (`*`) to match any app name.

```
<app_name>/
  models.py        ← (or models/ package) — matched by */models.py
  services.py      ← service layer (project convention)
  views.py         ← request handlers (maps to controllers key)
  migrations/      ← per-app migrations — matched by */migrations
  tests/
    unit/          ← unit tests — matched by */tests/unit
    integration/   ← integration tests — matched by */tests/integration
pyproject.toml     ← dependency manifest (PEP 517/518)
```

### Verification Tools

| Step | Command | Notes |
|------|---------|-------|
| test | `python manage.py test` | Django test runner (Unittest); `--tag <tag>` filters by tag |
| analyze | `ruff check .` | Fast Python linter (replaces flake8, isort, pyflakes) |
| format | `ruff format .` | Ruff's formatter (Black-compatible) |

### Agent Notes

- Path values are file globs, not directories. When navigating, expand `*/models.py` to find all `models.py` files across apps. An app may also use a `models/` package — check both.
- Django uses `views.py` for HTTP handlers; the `controllers` key maps to `*/views.py` to reflect this.
- `pyproject.toml` is the manifest (PEP 517/518); legacy projects may use `requirements.txt` alongside it.
- `--tag` filtering requires tests to be decorated with `@tag('tagname')`.

## express

**Language**: JavaScript | **Framework**: Express 5 | **Manifest**: `package.json`

### Directory Layout

```
src/
  models/          ← data models (Mongoose schemas, Sequelize models, etc.)
  services/        ← business logic
  controllers/     ← route handlers
migrations/        ← database migrations (Knex, Sequelize CLI, etc.)
test/
  unit/            ← unit tests
  integration/     ← integration / API tests
package.json       ← dependency manifest
```

### Verification Tools

| Step | Command | Notes |
|------|---------|-------|
| test | `npm test` | Delegates to the project's test script (Mocha, Jest, Vitest, etc.); pass `--grep <pattern>` for filtering |
| analyze | `npx eslint .` | Requires an ESLint config file in the project root (`eslint.config.js` or `.eslintrc.*`) |
| format | `npx prettier --write .` | Formats all files; respects `.prettierignore` and `.prettierrc` |

### Agent Notes

- Express imposes no directory structure. The layout above reflects common practice for larger apps; smaller projects may colocate routes, controllers, and models.
- `npm test` invokes whatever is in `package.json#scripts.test`. The actual test runner (and its filter flag) depends on the project. `--grep` is the Mocha convention; Jest uses `--testNamePattern` or `-t`.
- ESLint v9+ uses a flat config (`eslint.config.js`); older projects use `.eslintrc.json`. Agents should check which is present before assuming config paths.

## springboot

**Language**: Java | **Framework**: Spring Boot 3 | **Manifest**: `pom.xml`

### Directory Layout

Spring Boot follows Maven's standard layout. Package structure is nested under `src/main/java` with domain-based sub-packages.

```
src/
  main/
    java/
      com/example/         ← base package (varies per project)
        model/             ← matched by src/main/java/**/model
        service/           ← matched by src/main/java/**/service
        controller/        ← matched by src/main/java/**/controller
    resources/
      db/
        migration/         ← Flyway migrations (V1__description.sql)
  test/
    java/
      com/example/
        unit/              ← matched by src/test/java/**/unit
        integration/       ← matched by src/test/java/**/integration
pom.xml                    ← Maven dependency manifest
```

### Verification Tools

| Step | Command | Notes |
|------|---------|-------|
| test | `mvn test` | Maven Surefire plugin; use `-Dtest=ClassName` to run one class |
| analyze | `mvn checkstyle:check` | Checkstyle rules defined in `checkstyle.xml` (project root or `src/`) |
| format | `mvn spotless:apply` | Spotless Maven plugin; applies Google Java Format or custom rules |

### Agent Notes

- All path values use `**` glob wildcards because the actual package is project-specific (e.g. `com.acme.payments`). Agents should expand globs when navigating.
- Flyway migrations live in `src/main/resources/db/migration/` with prefix `V<version>__<description>.sql`.
- Spring Boot projects may split unit and integration tests differently from the schema above. If `unit/` or `integration/` directories don't exist, treat `src/test/java` as the test root.
- Gradle-based projects use the same source layout but replace `pom.xml` with `build.gradle` or `build.gradle.kts`. Custom profiles are the recommended approach for Gradle variants.

## go

**Language**: Go | **Framework**: stdlib (standard library) | **Manifest**: `go.mod`

### Directory Layout

Go projects following the standard layout (golang-standards/project-layout):

```
internal/
  domain/          ← domain types, entities, interfaces (models key)
  service/         ← business logic implementations
  handler/         ← HTTP handlers (controllers key; may also be api/ or http/)
migrations/        ← SQL migration files (golang-migrate convention)
internal/          ← tests live alongside source files (tests_unit key)
test/              ← end-to-end / integration tests (tests_feature key)
go.mod             ← module manifest
```

### Verification Tools

| Step | Command | Notes |
|------|---------|-------|
| test | `go test ./...` | Runs all tests recursively; use `-run <regex>` to filter |
| analyze | `golangci-lint run` | Aggregates many linters; config in `.golangci.yml` |
| format | `gofmt -w .` | Standard Go formatter; always produces canonical output |

### Agent Notes

- Go tests live in the same package as the code they test (`foo_test.go` next to `foo.go`). The `tests_unit` path (`internal/`) overlaps with source files — this is intentional. Agents should look for `*_test.go` files rather than separate test directories.
- `internal/` is a Go convention that restricts import visibility: only packages within the same module can import from `internal/`.
- Some projects use `pkg/` for exported packages and `cmd/` for main entry points. If present, `cmd/` is analogous to `controllers` for CLI apps.
- `golangci-lint` is a CI/CD tool and may not be installed locally. Run `go vet ./...` as a lightweight substitute.
- `gofmt -w .` is idempotent and safe to run at any time — it rewrites files in place to canonical format.

## Adding a Custom Profile

See the comment block at the top of `stacks:` in `forge.yaml` for a step-by-step guide. Key points:

1. Copy an existing profile block and rename the top-level key.
2. Ensure the six standard path keys are present (`models`, `services`, `controllers`, `migrations`, `tests_unit`, `tests_feature`).
3. All three verification commands are required (`test.command`, `analyze.command`, `format.command`).
4. For a service-local profile (one service only), add it to that service's `forge-service.yaml` under a `stacks:` key using the same structure.
